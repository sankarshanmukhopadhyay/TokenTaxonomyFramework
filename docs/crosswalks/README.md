# Crosswalks

These crosswalks position TTF artifacts against current tokenization standards, regulatory categories, and implementation patterns. They are not legal advice and do not convert TTF into a regulatory framework. They give maintainers and adopters a consistent way to record how a token artifact relates to external identifiers, lifecycle models, permissioned-transfer controls, and evidence expectations.

| Crosswalk | Purpose |
|---|---|
| [ISO Digital Token Identifier](iso-dti.md) | Align token artifact metadata with identifier assignment and registry references. |
| [MiCA Crypto-Asset Categories](mica.md) | Map TTF design questions to EU crypto-asset categories and issuer/holder concerns. |
| [IOSCO Financial Asset Tokenization](iosco-financial-asset-tokenization.md) | Translate market-integrity and investor-protection risks into artifact evidence requirements. |
| [BIS Tokenized Money and Unified Ledger](bis-tokenized-money.md) | Position tokenized money, tokenized deposits, settlement assets, and programmability. |
| [ERC-3643 Permissioned Tokens](erc-3643.md) | Map identity-aware transfer controls to TTF behaviors and metadata. |
| [LEI and vLEI Organizational Identity](lei-vlei.md) | Clarify issuer, verifier, custodian, registry, and smart-contract operator identity metadata. |
| [Lifecycle and CDM-Inspired Events](lifecycle-events.md) | Identify lifecycle events that should produce auditable evidence. |
